# 数据填写规范

## 总则

- 可求ai辅助，但要求文章必须经过人工审核，优先选择雅思、托福、bbc等权威机构音频、文章资料
- 字符编码：所有文本字段统一使用`UTF-8`，避免中文/特殊字符乱码； 
- 字段类型：严格匹配定义，数值型仅填数字、枚举型仅填指定值，无多余符号；
- 唯一性约束：`slug`（文章/标签）、`email`（用户）、联合主键（收藏/答题/阅读记录等）**严禁重复**； 
- 时间格式：所有时间字段统一为`YYYY-MM-DD HH:MM:SS`，未指定时可填`CURRENT_TIMESTAMP`； 
- 外键关联：`article_id`/`user_id`/`tag_id`等关联字段，必须存在于对应主表中，避免外键冲突。先填写**父表（用户表、文章表）**

## 各表数据需求细则

### questions, question_attempts, vocabulary_notes

此为功能扩展预留表，暂不进行填写

### speaking_attempts, reading_history, listening_progress, favorites, learning_statistics, 

此为自动处理表，暂不进行填写

### **articles：重点填写**

- 填写数量：**5个subject（具体哪个学科数据库内有写）各6-7篇**，总计30-35篇；
- 文章来源：严格审查文章，不能有语法拼写等错误，干脆使用**四六级真题、雅思真题**
- 难度分布：单subject内按难度**Easy:Intermediate:Advanced=2:2:2/3**均匀分配，保证各难度覆盖
- **这个表不储存文本内容，是个容器表**
- slug`: URL 友好的唯一标识，通常用小写字母和连字符组成（如 `bridge-design-intro`） 。
- audio_url: 指向存储中完整音频文件的有效链接 ，也就是说先要找到原始音频文件，然后这个属性存储他的url
- video_url：预留功能，暂时不做视频，先不填
- accent: 明确标注 'US'（美音）或 'UK'（英音），以便用户按偏好搜索 。
- total_duration: 以**秒**为单位的整数。必须与实际音频时长一致，用于计算资源完成率 。
- author, source：尽量填写1-2条，要求来源和作者必须正确可查，不能是ai生造的
- resource_type: 根据内容主体定义为 'text', 'audio' 或 'video' ，可以暂时不填
- word_count: 文章字数，填写数据时可以借助一些工具手动计算并填写，后续由后端算法执行

### **article_segments：重点填写**

- paragraph_index: 段落序号（从 0 或 1 开始）。用于在前端将句子重新组合成自然段落，支持“按段落显示原文” ，**在一开始要手动填写**
- sentence_index: 句子在段落内的序号。用于保持语序正确 。暂时不填。
- **`content_en`: 存入文章的单个段落，应包含标点符号，且首尾不要留多余空格。这是口语跟读的文本指纹 。**
- **content_cn: 对应单句的中文翻译。用于盲听模式下的“显文核对” ，先尝试填写3-5篇样例。**
- **`start_time` & `end_time`: 重点填写，表示每自然段在音频中的开始与结束位置，采用秒的形式，也就是第几秒开始第几秒结束，`start_time` 必须是该句在长音频中开始发音的时刻；`end_time` 则是发音结束时刻。两个时间点应严丝合缝，以保证“单句精听”循环时不突兀 。一开始填写时要手动输入，可以使用WhisperX这个工具辅助你分段**
